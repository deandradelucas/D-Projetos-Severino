## O BLOQUEIO DEFINITIVO (provado nesta branch)
A arquitetura por layers é elegante, mas **falha porque os arquivos não são "puros" de autoridade de cascata**.

**Prova — CTA "+ Nova transação" (dashboard, dark):**
- O dourado desejado vem de `04b` (`var(--hub-accent)`), que o squad mapeou em `base-overrides` (layer baixa).
- O neutro vem de `13-dashboard-neumorphic` (`skin-desktop`).
- Um gradiente **roxo** (`linear-gradient(135deg,#8b5cf6→#ff4fb8)`) está DENTRO de um arquivo `dark-theme` (layer `theme-dark`, alta).
- Na release (unlayered), os skins importados DEPOIS sobrescreviam o roxo do dark-theme → dourado vencia, via `!important`+ordem.
- Com layers: `theme-dark` > `skin-desktop` > `base-overrides` → **o roxo vence** (regressão). E mesmo removendo o roxo, o **neutro** (skin) venceria o **dourado** (base-overrides), pois base < skin.

**Conclusão:** o vencedor desejado (dourado) está num arquivo de layer BAIXA; os perdedores (roxo, neutro) em arquivos de layer ALTA. Nenhum mapeamento **arquivo→layer** replica a precedência real, porque os `!important` codificavam autoridade **por REGRA**, não por arquivo. O `dark-theme` precisa vencer em COR mas perder em `--primary`; isso é impossível com o arquivo inteiro numa única layer.

## O que seria necessário para concluir (caminho humano)
Uma de duas abordagens, ambas trabalho humano dedicado (dias):
1. **Extração rule-level:** mover regras individuais "autoritativas" (ex.: CTA dourado) para a layer `semantic`, caso a caso, para os ~27 conflitos mapeados + os que surgirem nas outras páginas/modais. Inviável de varrer por script; trivial no DevTools regra a regra.
2. **Refatoração na fonte (recomendada):** consolidar o estilo de cada componente num único lugar (eliminar a duplicação base-vs-skin-vs-dark que gera os conflitos), de modo que arquivo = autoridade. Aí o `@layer` funciona limpo. É reescrever o design system por componente.
