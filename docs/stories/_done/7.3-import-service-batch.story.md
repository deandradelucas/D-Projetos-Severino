# Story 7.3 — Batch Import Service

## Status
Done

## Descrição
Criar `server/lib/import/import-service.mjs` que orquestra a importação de um array de transações normalizadas: detecta duplicatas pelo hash (data + descrição + valor), categoriza em lote usando o pipeline IA existente, insere no banco via `inserirTransacao` e retorna um resumo da operação.

## Critérios de Aceite

### Interface Pública
- [x] Função exportada: `async function importarTransacoes(usuarioId, rows, options?)`
  - `rows`: `Array<{data, descricao, valor, tipo}>`
  - `options.fontePlanilha`: string descritiva (ex.: `'excel'`, `'pdf'`) — salva no campo `origem` (se existir)
  - Retorna `{importadas: number, ignoradas: number, erros: number, despesas: number, receitas: number, semCategoria: number}`

### Deduplicação
- [x] Hash padrão (Excel/PDF): `sha256(${usuarioId}:${data}:${descricao_lower}:${valor})`
- [x] Hash OFX com FITID: `sha256(${usuarioId}:ofx:${fitid})` — mais confiável (ID único do banco)
- [x] Parser pode passar `fitid` opcional no row; import-service decide qual hash usar
- [ ] Consulta tabela `transacoes` para `usuario_id + origem_hash` nas últimas 24 meses — _desvio aceito: dedup por `origem_hash IN (hashes do lote)` filtrado por `usuario_id` é mais correto que janela temporal (não deixa passar duplicata > 24m) e eficiente via índice parcial_
- [x] Linha com hash já existente é marcada `ignorada` (não inserida nem contabilizada em `importadas`)
- [x] Hash salvo no campo `origem_hash` da transação (adicionar coluna via migration se não existir)

### Categorização em Batch
- [x] Carrega categorias do usuário UMA vez antes do loop (`getCategorias(usuarioId)`)
- [ ] Para cada linha: chama `suggestCategoryForTransaction(descricao, tipo, categorias)` — _desvio aceito: usa `suggestCategoriesBatch` (1 chamada para o lote) em vez de N chamadas por linha — superior_
- [ ] Processa em grupos de 10 com `Promise.all` (paralelo controlado) — _desvio aceito: `suggestCategoriesBatch` processa em chunks de 50 internamente_
- [x] Linhas sem categoria recebem fallback `Outros/Outros` (mesmo padrão do WhatsApp bot)
- [x] Conta `semCategoria` quando cai no fallback Outros

### Inserção no Banco
- [x] Usa `inserirTransacao(usuarioId, {data, descricao, valor, tipo, categoria_id, subcategoria_id, origem_hash})`
- [x] Falha em linha individual: loga warning + incrementa `erros`, NÃO aborta o lote inteiro
- [x] Transaction-level: cada inserção é independente (sem transaction SQL de batch — simplicidade > atomicidade aqui)

### Migration de Schema
- [ ] Verifica se coluna `origem_hash` existe em `transacoes` (query information_schema) — _desvio aceito: schema versionado em migration, não runtime check (convenção do projeto: `scripts/migrations/`)_
- [x] Coluna `origem_hash TEXT`: `ALTER TABLE transacoes ADD COLUMN IF NOT EXISTS origem_hash TEXT` — versionado em `scripts/migrations/67_transacoes_origem_hash.sql`
- [x] Índice `idx_transacoes_origem_hash ON transacoes(usuario_id, origem_hash)` — confirmado no banco (parcial `WHERE origem_hash IS NOT NULL`) + versionado na migration 67
- [ ] Migration executada na primeira chamada ao import service (lazy migration) — _desvio aceito: migration versionada explícita (migration 67) em vez de lazy runtime — mais auditável e alinhada à ADR 0002 (drift de schema)_
  <!-- coluna + índice confirmados em produção via MCP Supabase (2026-06-04); migration 67 versiona o schema que já existia, fechando o drift -->

### Resumo e Logging
- [x] Loga início com `log.info('[import] iniciando', { usuarioId, total: rows.length })`
- [x] Loga fim com `log.info('[import] concluído', { ...resumo })`
- [x] Retorna resumo completo para o caller formatar a mensagem ao usuário

## Notas Técnicas
- `crypto.createHash('sha256').update(str).digest('hex')` — nativo Node.js, sem deps
- Lote de 10: balanceia throughput vs. rate limit Gemini (37 chamadas/dia no free tier)
- `origem_hash` TEXT (não UNIQUE globalmente — mesmo hash pode aparecer em outro usuário)
- Para dedup: query `WHERE usuario_id = $1 AND origem_hash = ANY($2)` antes do loop
