# Story 7.4 — WhatsApp Document Handler

## Status
InReview

## Descrição
Modificar `whatsapp-evolution-inbound.mjs` para detectar mensagens de documento (.xlsx, .xls, .pdf) enviadas pelo WhatsApp, baixar o arquivo via Evolution API (mesmo endpoint já usado para áudio), rotear para o import service e responder ao usuário com um resumo da importação.

## Critérios de Aceite

### Detecção de Documento no Webhook
- [x] Em `buildBotBodyFromEvolutionSingle`: detectar `msg.documentMessage` (ou `msg.documentWithCaptionMessage`)
- [x] Extrair `mimeType` de `msg.documentMessage.mimetype`
- [x] Mimetypes aceitos: `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`, `application/vnd.ms-excel`, `text/csv`, `application/pdf`, `application/x-ofx`, `application/ofx`
- [x] Extensões como fallback se mimetype for genérico (`application/octet-stream`): `.xlsx`, `.xls`, `.csv`, `.pdf`, `.ofx`, `.qfx` — extrair do `msg.documentMessage.fileName`
- [x] Se documento detectado: incluir no bot body `{documentMessageId, documentMimeType}` em vez de `audioUrl/audioBase64`
- [x] Documentos com mimetype não-suportado: retornar `null` (ignorar, não processar)

### Download do Arquivo
- [ ] Nova função `documentBufferFromEvolutionMedia(body)` em `whatsapp-evolution-inbound.mjs`
- [x] Reutiliza `buildGetBase64MessagePayload(body)` e a lógica de `audioBufferFromEvolutionMedia`
- [x] Chama `/chat/getBase64FromMediaMessage/{instance}` com o `documentMessageId`
- [x] Converte base64 → Buffer via `base64ToBuffer`
- [ ] Timeout: 60s (documentos maiores que áudio)
- [x] Falha de download: retorna reply `'📄 Não consegui receber o arquivo. Tente novamente.'`

### Roteamento no processWhatsappBotBody
- [x] Detectar `body.documentMessageId` como novo `inputType = 'document'`
- [x] Verificação de tamanho antes de processar:
  - xlsx/xls/csv/ofx/qfx: rejeita > 10MB com mensagem `'📊 Arquivo muito grande (máx. 10MB). Compacte e tente novamente.'`
  - pdf: rejeita > 20MB com mensagem `'📄 PDF muito grande (máx. 20MB). Tente um extrato menor.'`
- [x] Chamar `processarImportacaoDocumento(phone, documentBuffer, mimeType, fileName)` (nova função)

### Função processarImportacaoDocumento
- [x] Localização: `server/lib/domain/whatsapp-import.mjs` (novo arquivo)
- [x] Busca usuário pelo telefone (`buscarUsuarioPorTelefone`)
- [x] Verifica assinatura ativa com `computeAssinaturaFlags` — trial também pode importar
- [x] Roteia parser conforme mimeType/extensão:
  - xlsx/xls/csv → `parseExcelTransactions`
  - pdf → `parsePdfTransactions`
  - ofx/qfx → `parseOfxTransactions`
- [x] Detecção por extensão do `fileName` como fallback quando mimeType é `application/octet-stream`
- [x] Em caso de erro do parser (FORMATO_INVALIDO, PLANILHA_VAZIA, etc.): retorna mensagem amigável
- [x] Chama `importarTransacoes(usuarioId, rows, { fontePlanilha: 'whatsapp' })`
- [x] Formata reply com resumo da importação

### Mensagens de Resposta
- [x] Sucesso:
  ```
  ✅ Importei *{N} transações* com sucesso!
  
  💰 {despesas} despesas · 🟢 {receitas} receitas
  {semCategoria > 0 ? `⚠️ {semCategoria} sem categoria — aparecem como "Outros"` : ''}
  
  Abra o app para conferir e ajustar as categorias. 📱
  ```
- [x] Nenhuma transação encontrada: `'📄 Não encontrei transações válidas no arquivo. Verifique se é um extrato bancário padrão.'`
- [x] Arquivo inválido: `'❌ Formato não reconhecido. Envie um arquivo .xlsx, .xls, .csv ou .pdf.'`
- [x] Usuário não cadastrado: mesmo fluxo já existente (`'📱 Número não cadastrado...'`)

### Logging e Monitoramento
- [x] `log.info('[whatsapp-import] documento recebido', { phone, mimeType, sizeKb })`
- [x] `log.info('[whatsapp-import] concluído', { phone, ...resumo })`
- [x] Erros de parse/import: `log.warn` com detalhes, NÃO 500 para o webhook

## Notas Técnicas
- Evolution API: `documentMessage` tem os mesmos campos de mídia que `audioMessage` (`url`, `base64`, `mediaUrl`)
- O endpoint `/chat/getBase64FromMediaMessage` aceita qualquer tipo de mídia — funciona para docs
- `documentWithCaptionMessage` é um wrapper — já temos `unwrapInnerMessage` que o desembrulha
- Não é necessário `convertToMp4: false` para documentos (campo ignorado pela API)
