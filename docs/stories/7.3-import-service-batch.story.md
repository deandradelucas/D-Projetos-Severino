# Story 7.3 — Batch Import Service

## Status
Draft

## Descrição
Criar `server/lib/import/import-service.mjs` que orquestra a importação de um array de transações normalizadas: detecta duplicatas pelo hash (data + descrição + valor), categoriza em lote usando o pipeline IA existente, insere no banco via `inserirTransacao` e retorna um resumo da operação.

## Critérios de Aceite

### Interface Pública
- [ ] Função exportada: `async function importarTransacoes(usuarioId, rows, options?)`
  - `rows`: `Array<{data, descricao, valor, tipo}>`
  - `options.fontePlanilha`: string descritiva (ex.: `'excel'`, `'pdf'`) — salva no campo `origem` (se existir)
  - Retorna `{importadas: number, ignoradas: number, erros: number, despesas: number, receitas: number, semCategoria: number}`

### Deduplicação
- [ ] Hash padrão (Excel/PDF): `sha256(${usuarioId}:${data}:${descricao_lower}:${valor})`
- [ ] Hash OFX com FITID: `sha256(${usuarioId}:ofx:${fitid})` — mais confiável (ID único do banco)
- [ ] Parser pode passar `fitid` opcional no row; import-service decide qual hash usar
- [ ] Consulta tabela `transacoes` para `usuario_id + origem_hash` nas últimas 24 meses
- [ ] Linha com hash já existente é marcada `ignorada` (não inserida nem contabilizada em `importadas`)
- [ ] Hash salvo no campo `origem_hash` da transação (adicionar coluna via migration se não existir)

### Categorização em Batch
- [ ] Carrega categorias do usuário UMA vez antes do loop (`getCategorias(usuarioId)`)
- [ ] Para cada linha: chama `suggestCategoryForTransaction(descricao, tipo, categorias)`
- [ ] Processa em grupos de 10 com `Promise.all` (paralelo controlado)
- [ ] Linhas sem categoria recebem fallback `Outros/Outros` (mesmo padrão do WhatsApp bot)
- [ ] Conta `semCategoria` quando cai no fallback Outros

### Inserção no Banco
- [ ] Usa `inserirTransacao(usuarioId, {data, descricao, valor, tipo, categoria_id, subcategoria_id, origem_hash})`
- [ ] Falha em linha individual: loga warning + incrementa `erros`, NÃO aborta o lote inteiro
- [ ] Transaction-level: cada inserção é independente (sem transaction SQL de batch — simplicidade > atomicidade aqui)

### Migration de Schema
- [ ] Verifica se coluna `origem_hash` existe em `transacoes` (query information_schema)
- [ ] Se não existir: aplica migration `ALTER TABLE transacoes ADD COLUMN IF NOT EXISTS origem_hash TEXT`
- [ ] Índice: `CREATE INDEX IF NOT EXISTS idx_transacoes_origem_hash ON transacoes(usuario_id, origem_hash)`
- [ ] Migration executada na primeira chamada ao import service (lazy migration)

### Resumo e Logging
- [ ] Loga início com `log.info('[import] iniciando', { usuarioId, total: rows.length })`
- [ ] Loga fim com `log.info('[import] concluído', { ...resumo })`
- [ ] Retorna resumo completo para o caller formatar a mensagem ao usuário

## Notas Técnicas
- `crypto.createHash('sha256').update(str).digest('hex')` — nativo Node.js, sem deps
- Lote de 10: balanceia throughput vs. rate limit Gemini (37 chamadas/dia no free tier)
- `origem_hash` TEXT (não UNIQUE globalmente — mesmo hash pode aparecer em outro usuário)
- Para dedup: query `WHERE usuario_id = $1 AND origem_hash = ANY($2)` antes do loop
